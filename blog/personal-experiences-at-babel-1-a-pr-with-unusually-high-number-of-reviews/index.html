Personal Experiences at Babel #1 ‚Äî A PR with Unusually High Number of Reviews

August 11, 2017 ¬∑ 6 min read
Peeyush Kushwaha

We landed the parser support for the stage-2 decorators spec last week at Babylon ‚Äî the parser for Babel. If you don‚Äôt know what a decorator is, the gist of it is that a decorator gives some concise syntax to affect the definition of a class or a class method which you decorate.

JavaScript

  @configurable(false)
  @enumerable(true)
  method() {}

  @throttle(500)
  expensiveMethod() {}
}

One of the more remarkable things about this PR was the number of reviews it received

Perhaps this could be because decorators really are one of the much hyped about features in JavaScript. Angular even considered making their own JS flavor called AtScript before they decided to go with TypeScript since they love decorators (or as they liked to call it ‚Äúannotations‚Äù) so much.

Well, there is more to the story. As I was recently discussing with a mentor, reviewing PRs is a tough job. Reviewing PRs is comparably as hard as solving the bug in the first place was. Apart from the technical aspect of reviewing ‚Äî which is ensuring that the bug is being fixed optimally (by perhaps even getting an idea of how they would solve the issue and seeing how the submitted patch compares to their idea) ‚Äî there‚Äôs another big hindrance. A reviewer has to be aware of the whole issue, the discussion surrounding it, and have familiarity with the part of the codebase that the PR makes changes to.1

When I initially joined Babel, and was not-so-familiar with the codebase, every issue I encountered was almost instantly answered after I posted it in Babel‚Äôs chatroom, which left me with the (wrong) impression that perhaps the maintainers are some god-like figures who know it all and that everyone‚Äôs expected to adhere to the same fictitious standards.

Even after becoming familiar with the codebase, I was submitting PRs without proper documentation under the assumption that it took me a while to solve the issue and see all things, but if the reviewers see the code they‚Äôd instantly be able to evaluate it just like they were answering my questions.

Eh! Very wrong. Let me just bust this myth (assuming I‚Äôm not the only one who has felt it). Even they (maintainers) won‚Äôt have all the answers at times, and sometimes you‚Äôll have to search for yourself ‚Äî and that‚Äôs how it should be.2

In open source, there are a lot of people who want to contribute, but are unable to because they don‚Äôt know how to code / they don‚Äôt know how to present PRs / they don‚Äôt know what the project wants / they don‚Äôt know what the maintainers want / a ton of other things. A lot of times you‚Äôll find help along the way, but a lot of that is controlled by factors beyond your control.3

One of the joys of getting your PR merged is not just the programming but somehow making the project move forward in the way it is expected to be moved forward. And doing that by somehow identifying what the project needs and being able to deliver that.

In order to merge this PR I had to find people and talk to them ‚Äî people who use decorators, people who are interested in seeing an implementation of decorators, people who want to contribute to babel for decorators. After getting consensus on how to move forward4, I had to go through the spec and all the existing discussions surrounding it so that my understanding of the spec could be up to speed with other people.

And finally ‚Äî the most important part which I think got this PR the number of reviews that we saw ‚Äî making it easy for people who‚Äôll be reviewing my PR by explaining everything they would need to get up to speed with the whole situation. By chance, at the time the PR I made was able to satisfy some of the criteria I mentioned earlier:

  1. Making sure reviewers are aware of the whole issue (by mentioning in detail the decisions I‚Äôve taken so they don‚Äôt necessarily have to look at the code to figure it out)
  2. The discussion surrounding it (by mentioning alternate viewpoints on some of the decisions so as to make it easier to compare them with the decisions made)
  3. Explaining clearly my strategy to solve the problem (to assist the technical aspect of reviewing ‚Äî so that the reviewers can know what I‚Äôve done and then see the code rather than the other way around)

And that‚Äôs what did it! (or so I think). So there‚Äôs the mystery unraveled ‚Äî A PR with unusually high number of reviews¬π.

Also note that if you‚Äôre looking for decorators support in Babel, we still have a long way to go.
